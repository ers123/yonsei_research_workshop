## 문제: AI는 매번 처음부터 다시 읽는다

AI 코딩이나 연구에 AI를 써 본 사람이라면 한 번쯤 이런 경험이 있을 겁니다. 분명히 아까 설명했는데 AI가 또 같은 걸 물어봅니다. 이전에 분석한 내용을 완전히 잊어버리고 전혀 다른 방식으로 새로 시작합니다.

이게 바로 **컨텍스트 문제**입니다. 컨텍스트란 AI가 한 번에 기억할 수 있는 정보의 양 — 쉽게 말하면 AI의 단기 기억력입니다. 그리고 이 기억 용량에는 한계가 있습니다.

비유하면 이렇습니다: 매일 아침 기억이 리셋되는 신입 사원이 있다고 상상해보세요. 어제 했던 업무, 프로젝트 히스토리, 회의록을 전부 까먹은 상태로 출근합니다. 매일 서류함의 문서 100개를 처음부터 다시 읽어야 합니다. 똑똑하긴 한데, 매번 처음부터 다시 시작하니까 업무 효율이 바닥입니다. **지금 AI가 정확히 이 상황입니다.**

기존의 RAG(Retrieval-Augmented Generation) 방식은 질문할 때마다 도서관에 가서 책을 찾아 읽는 것과 같습니다. 논문 30편을 업로드하면, 매번 30편을 전부 훑습니다. 다음 질문에도 또 30편을 전부 훑습니다.

## 해결: Karpathy의 LLM Wiki 패턴

Andrej Karpathy — 전 Tesla AI Director, OpenAI 창립 멤버 — 가 2026년 4월 제안한 개념입니다. 아이디어 자체는 놀라울 정도로 단순합니다:

> **AI에게 원본 문서를 매번 읽히지 말고, AI가 미리 핵심만 정리한 위키를 만들어서 이후에는 그 위키만 참조하게 하자.**

Karpathy의 표현을 빌리면: *"지식이 컴파일된 상태로 유지되기 때문에, 매번 처음부터 다시 추출할 필요가 없다."* 위키는 지속적으로 축적되는 산출물입니다. 교차 참조가 이미 구축되어 있습니다.

**실측 데이터** (LLM Wiki Compiler 플러그인 기준):
- 마크다운 파일 383개 (13.1MB) → 위키 13개 파일로 압축. **81배 압축.**
- 회의 녹취록 130개 (122,625줄) → 요약 1개 파일 (244줄). **503배 압축.**
- 세션 시작 시 컨텍스트: 47,000 토큰 → 7,700 토큰. **84% 절감.**
- 질문당 리서치 토큰: 8,000 → 600. **역시 84% 절감.**

## 3계층 구조

```
my-research-wiki/
│
├── raw/                  ← 1계층: 원본 소스 (불변)
│   ├── paper_kim_2025.pdf
│   ├── interview_participant_03.md
│   ├── conference_notes_2026.md
│   └── ...
│
├── wiki/                 ← 2계층: AI가 정리한 위키 페이지
│   ├── index.md              전체 목록 + 한 줄 요약
│   ├── literature_review.md  선행연구 요약 비교표
│   ├── methodology.md        연구 방법론 비교 분석
│   ├── key_findings.md       주요 발견 교차 비교
│   ├── gaps.md               연구 공백 분석
│   ├── theoretical_framework.md  이론적 프레임워크
│   └── log.md                작업 기록 (시간순)
│
└── CLAUDE.md             ← 3계층: 스키마 (위키의 규칙과 구조)
```

**1계층 — 원본 소스 (raw/)**: 도서관의 책들. 논문 PDF, 인터뷰 녹취록, 회의록, 강의 노트 등 날 것 그대로의 자료. 양이 방대하고 정리가 안 되어 있습니다. 이 계층은 불변(immutable)입니다 — 한 번 넣으면 수정하지 않습니다.

**2계층 — 위키 (wiki/)**: 도서관 책들을 읽고 핵심만 정리한 요약 노트. AI가 원본 자료를 읽고 중요한 내용만 추려서 주제별로 깔끔하게 정리해 놓은 것입니다. 사람이 읽기에도, AI가 읽기에도 최적화되어 있습니다.

**3계층 — 스키마 (CLAUDE.md)**: 목차와 규칙. 위키를 어떤 구조로 만들지, 어떤 형식을 따를지 정해 놓은 설정 파일입니다. 워크숍에서 다룬 "프로젝트 맥락 문서"가 바로 이것입니다.

## 3가지 핵심 작업

### 1. Ingest (소화) — 새 자료 추가

새로운 문서가 들어오면 AI가 그 문서를 읽고 핵심을 추출합니다. 그리고 관련된 위키 페이지들을 찾아서 업데이트합니다. 하나의 문서가 들어오면 보통 10~15개의 위키 페이지에 영향을 줍니다.

```
> raw/ 폴더에 새로 추가된 paper_lee_2026.md를 읽고,
> CLAUDE.md의 스키마에 따라 관련 위키 페이지들을 업데이트해줘.
> 변경 사항을 log.md에 기록해줘.
```

### 2. Query (질문) — 위키 기반 답변

누군가 질문을 하면 AI가 원본 문서 대신 위키에서 답을 찾아 알려줍니다. 여기서 진짜 멋진 부분: **좋은 답변이 나오면 그 답변을 다시 위키에 저장합니다.** Karpathy는 이를 "지식 복리(knowledge compounding)"라고 표현했습니다. 질문과 답변을 통해 위키의 지식이 계속 풍부해지는 겁니다.

```
> 내 연구에서 사용할 수 있는 이론적 프레임워크를 
> wiki/의 내용을 바탕으로 3가지 제안해줘.
> 각각의 근거와 한계를 포함해줘.
```

### 3. Lint (점검) — 품질 유지

주기적으로 AI가 위키를 점검합니다: 모순되는 내용이 없는지, 오래된 정보가 없는지, 연결이 끊어진 고아 페이지가 없는지 확인합니다. 도서관 사서가 정기적으로 서가를 정리하는 것과 같습니다.

```
> wiki/ 전체를 점검해줘:
> 1. 페이지 간 모순되는 내용
> 2. 출처가 누락된 주장
> 3. 교차 참조가 빠진 관련 내용
> 4. 최근 추가 자료가 반영되지 않은 페이지
> 결과를 lint_report.md로 작성해줘.
```

## 대학원 연구에 적용하기: 15분 셋업

### 준비물

- **Obsidian** (obsidian.md) — 무료 마크다운 에디터. 파일 간 링크와 그래프 뷰 지원
- **Obsidian Web Clipper** — 브라우저 확장. 웹 페이지/논문을 마크다운으로 변환
- **AI 도구** — Claude(claude.ai), Claude Code, 또는 로컬 AI (별첨 1 참조)

### 설정 방법

**1단계: Obsidian Vault 생성 (2분)**

Obsidian 설치 → "Create new vault" → 이름: `thesis-wiki` (또는 원하는 이름)

**2단계: 폴더 구조 생성 (1분)**

Vault 안에 두 폴더를 만듭니다:
- `raw/` — 원본 자료 저장소
- `wiki/` — AI가 생성/관리할 위키 페이지

**3단계: CLAUDE.md 스키마 작성 (10분)**

Vault 루트에 `CLAUDE.md` 파일을 만들고, 아래를 본인 연구에 맞게 수정합니다:

```markdown
# 논문 위키 스키마

## 목적
이 위키는 [연구 주제]에 대한 [석사/박사] 논문 준비를 위한 지식 베이스이다.

## 위키 페이지 구조
각 페이지는 아래 형식을 따른다:

### 필수 페이지
- index.md: 전체 소스 목록 + 한 줄 요약. 카테고리별 정리.
- literature_review.md: 선행연구 비교표 (저자, 연도, 방법론, 주요 발견, 한계)
- methodology.md: 연구 방법론 비교 분석 + 본 연구 적용 가능성
- theoretical_framework.md: 이론적 프레임워크 후보 + 근거
- gaps_and_opportunities.md: 기존 연구의 공백 + 본 연구의 기여점
- my_contribution.md: 본 연구의 차별성, NTIS 차별성 검토 결과 반영
- timeline.md: 연구 일정, 마일스톤, 지도교수 피드백 기록

### 선택 페이지 (필요 시 추가)
- participants.md: 연구 참여자 프로필 (질적 연구)
- instruments.md: 측정 도구/설문지 비교
- ethics.md: IRB, 윤리적 고려사항
- glossary.md: 핵심 용어 정의

## 규칙
1. 모든 주장에는 출처를 표기한다 (저자, 연도)
2. 위키 페이지 간 [[상호 참조]]를 적극 활용한다
3. 새 자료 추가 시 관련 페이지를 모두 업데이트한다
4. 상충하는 발견은 삭제하지 말고 병기하며 비교한다
5. log.md에 모든 변경 사항을 시간순으로 기록한다
6. 불확실한 내용은 [미확인] 태그를 붙인다

## 출력 형식
- 한국어로 작성
- 학술적 문체 유지
- 표(table)를 적극 활용하여 비교 정리
```

**4단계: 원본 자료 추가 (2분~)**

`raw/` 폴더에 논문, 인터뷰 녹취록, 메모 등을 마크다운으로 변환하여 저장합니다.
- PDF 논문 → Obsidian Web Clipper 또는 수동 복사
- 웹 페이지 → Web Clipper로 클리핑
- 인터뷰 녹취 → 텍스트 파일 저장

**5단계: 위키 초기 생성**

AI에게 다음과 같이 요청합니다:

```
raw/ 폴더의 모든 자료를 읽고, 
CLAUDE.md 스키마에 따라 wiki/ 페이지를 생성해줘.
각 페이지에 출처를 명시하고, 페이지 간 교차 참조를 연결해줘.
index.md와 log.md도 함께 생성해줘.
```

## 왜 NotebookLM이나 매번 질문하기보다 나은가

| | 매번 AI에 질문 (RAG) | NotebookLM | LLM Wiki |
|---|---|---|---|
| 토큰/비용 | 매번 원본 재탐색 (낭비) | Google 서버 처리 | 위키만 참조 (84%↓) |
| 지식 축적 | 없음 (세션 끝나면 리셋) | 제한적 | 복리로 성장 (질문할수록 풍부) |
| 데이터 소유 | 클라우드 의존 | Google 서버 | 내 PC, plain markdown |
| 커스터마이징 | 불가 | 제한적 | 스키마로 완전 자유 |
| 벤더 종속 | 있음 | Google 의존 | 없음 (마크다운 파일) |
| 오프라인 | 불가 | 불가 | 로컬 AI와 조합 시 가능 |

핵심 차이: **LLM Wiki에서는 지식이 축적됩니다.** 질문할 때마다 위키가 더 풍부해지고, 다음 질문에 더 좋은 답을 줍니다. NotebookLM이나 ChatGPT에 매번 질문하는 것은 매번 새로 시작하는 것과 같습니다.

## 실전 워크플로 예시: 석사 논문 준비

### 1주차: 기반 구축
- Obsidian Vault 생성 + 스키마 작성
- 핵심 논문 10편을 `raw/`에 추가
- AI에게 위키 초기 생성 요청 → `wiki/` 6~8개 페이지 자동 생성

### 2~4주차: 자료 축적
- 매주 논문 5~10편 추가 → AI가 ingest하여 위키 자동 업데이트
- 세미나, 지도교수 미팅 메모도 `raw/`에 추가
- 주 1회 lint 실행 → 모순/누락 점검

### 5~8주차: 분석 심화
- "기존 연구의 방법론적 한계를 정리해줘" → 위키 기반 즉시 답변
- "내 연구의 차별성을 3가지로 정리해줘" → gaps.md + my_contribution.md 종합
- NTIS 차별성 검토 결과도 위키에 반영

### 9주차~: 집필
- "서론의 논리 구조를 literature_review.md와 gaps.md를 바탕으로 제안해줘"
- 위키가 이미 정리되어 있으므로, 집필이 구조화된 자료에서 시작됨
- 새 발견이나 수정사항은 위키에 반영 → 논문과 위키가 동시 진화

## 주의사항

1. **위키는 출발점이지 결론이 아닙니다** — AI가 정리한 위키를 맹신하지 마세요. 원본 확인은 필수입니다.
2. **스키마를 자주 다듬으세요** — 연구가 진행되면서 필요한 페이지 구조가 바뀝니다. CLAUDE.md를 정기적으로 업데이트하세요.
3. **민감 데이터는 로컬에서** — IRB 데이터가 포함된 위키는 반드시 로컬 AI(Ollama)와 조합하세요. 별첨 1을 참조하세요.
4. **Git으로 버전 관리 권장** — 위키 폴더를 Git으로 관리하면 변경 이력 추적이 가능합니다.
