# 별첨 2: 나만의 LLM Wiki 만들기
**Obsidian + Claude Code 로 "AI 가 읽을 수 있는 개인 지식 베이스" 구축하기**

> 대상: 논문을 많이 읽지만 "읽고 잊어버리는" 문제를 겪는 문과·사회과학 대학원생.
> 목표: 오늘 저녁 안에 나만의 위키 Vault 를 만들고, 첫 5개 지식 항목을 채워본다.
> 원본 아이디어: Andrej Karpathy, "LLM Wiki" (gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)

---

## 왜 이런 걸 만드나

### 문제 1: AI 는 매 대화마다 "초면"

ChatGPT 나 Claude 에게 어제 설명했던 내 연구 주제를 오늘 다시 설명하고 있는 자기 자신을 본 적이 있을 겁니다. AI 의 "단기 기억"은 대화창 하나가 끝나면 사라집니다. 매번 설명을 반복하면 시간 낭비인 것은 물론, **AI 가 내 연구의 전체 지도를 절대 갖지 못합니다.**

### 문제 2: 나도 내가 읽은 걸 잊는다

3개월 전 읽은 논문에서 "검증부채(verification debt)" 개념이 인상적이었다고 메모해뒀는데, 오늘 서론을 쓰다가 그 개념을 다시 찾으려니 어느 논문이었는지 기억이 안 납니다. Zotero 에 있긴 한데 어느 PDF 에 하이라이트를 쳤는지는 모릅니다.

### 해결: "AI 가 읽을 수 있는 내 두뇌의 외장"

아이디어는 단순합니다:

1. **내가 읽은 것의 요약**을 마크다운 파일로 차곡차곡 쌓는다.
2. 그 파일들이 **서로 링크**로 연결된다 (위키처럼).
3. AI 에게 매번 "이 폴더를 먼저 읽어라" 고 시킨다.

결과: **나도 AI 도 같은 지식 지도를 공유한다.** 내가 기억 못 해도 AI 가 기억한다. AI 가 멋대로 지어내도 내가 근거를 확인한다.

## 왜 Obsidian 인가

이 가이드는 **Obsidian** 을 기본 도구로 씁니다. 이유:

- **무료·로컬** — 구독비 없음, 파일이 내 PC 에만 저장됨 (민감한 연구 메모 안전)
- **마크다운** — 파일이 그냥 `.md` 텍스트 파일. Obsidian 이 망해도 메모는 남음
- **[[위키 링크]]** — `[[개념명]]` 만 쓰면 자동으로 페이지끼리 연결됨. 한 번 써보면 못 끊음
- **Claude Code 친화** — 그냥 폴더이므로 AI 가 쉽게 읽고 쓸 수 있음
- **플러그인 생태계** — 논문 연동, 그래프 시각화, 달력 등 확장 풍부

Notion 을 이미 쓰고 있다면? Notion 도 가능합니다만, (a) 파일이 Notion 서버에 있고, (b) AI 가 직접 읽기 어렵고, (c) 오프라인·검색 속도에서 불리합니다. **이 워크숍에서는 Obsidian 을 권장하고, Notion 은 "이미 쓰고 있다면 병행" 정도로 봅니다.**

## Step 1 — Obsidian 설치

### 1-1. macOS

1. 브라우저에서 **https://obsidian.md** 접속
2. "Download" → "Download for macOS (Apple Silicon)" 또는 Intel 선택
3. `Obsidian-x.x.x.dmg` 다운로드 → 더블클릭 → Obsidian 아이콘을 Applications 폴더로 드래그
4. Launchpad 에서 Obsidian 실행 → 처음 한 번 "열겠습니까?" 묻는 macOS 보안 대화상자에서 "열기" 클릭

### 1-2. Windows

1. 브라우저에서 **https://obsidian.md** 접속
2. "Download for Windows" → `Obsidian.x.x.x.exe` 다운로드
3. 설치 파일 실행 → Next 계속 → 설치 완료 시 자동 실행

⚠️ **설치 직후 화면**: Obsidian 이 처음 열리면 검정색 배경의 환영 화면이 나옵니다. 아직 "내 노트 폴더(Vault)" 가 없는 상태이므로 화면이 허전해 보입니다. 다음 Step 에서 만듭니다.

### ✔ 체크포인트

- [ ] Obsidian 이 실행되고 검정 배경의 환영 화면이 보인다
- [ ] Obsidian 앱 좌상단 메뉴가 작동한다 (파일·편집·보기 등)

## Step 2 — Vault 만들기 & 폴더 구조 세팅

Obsidian 에서 "Vault(금고)" 는 그냥 **하나의 폴더** 입니다. 그 폴더 안에 마크다운 파일이 쌓입니다.

### 2-1. 새 Vault 생성

환영 화면에서:

1. **"Create new vault"** 클릭
2. Vault name: `research-wiki` (원하는 이름 OK)
3. Location: 편한 곳 지정. 권장 위치는 아래.
   - macOS: `/Users/<본인아이디>/Documents/research-wiki`
   - Windows: `C:\Users\<본인아이디>\Documents\research-wiki`
4. "Create" 클릭

왼쪽에 빈 파일 탐색기가 뜨면 성공입니다.

### 2-2. 폴더 구조 만들기

왼쪽 파일 탐색기에서 **우클릭 → "New folder"** 를 반복해 다음 구조를 만듭니다:

```
research-wiki/
├── raw/           ← 원본 자료 (논문 PDF 텍스트, 녹취록, 스크랩)
├── wiki/          ← 내가 쓰는 지식 항목 (개념·방법론·이론)
│   ├── concepts/      개념 (예: 검증부채, 디지털 격차)
│   ├── methods/       방법론 (예: 근거이론, PRISMA)
│   └── authors/       인물·저자 (예: Karpathy, 조한혜정)
├── papers/        ← 논문 1편당 1개 파일 (요약 + 인용)
├── drafts/        ← 내 글 초안 (서론·문헌검토 등)
└── log/           ← 날짜별 작업 기록 (journal)
```

⚠️ **첫인상 팁**: Obsidian 은 폴더 계층이 깊어지면 오히려 관리가 어려워집니다. 위 5개 정도의 **최상위 폴더** 면 충분합니다. 안에서 더 나누는 건 나중에 필요할 때 합니다.

### 2-3. 첫 시각 점검

왼쪽 상단의 **파일 탐색기 아이콘**(네모 2개) 을 눌러 위 폴더들이 모두 보이는지 확인하세요. 그리고 왼쪽 하단 ⚙️(설정) 아이콘 → **Appearance** → Theme 을 편한 것으로 바꿔두면 이후 작업이 쾌적합니다.

### ✔ 체크포인트

- [ ] `research-wiki` Vault 가 생성되고 Obsidian 왼쪽에 폴더 트리가 보인다
- [ ] `raw/`, `wiki/`, `papers/`, `drafts/`, `log/` 가 모두 있다
- [ ] Finder(macOS) 또는 파일 탐색기(Windows) 로 해당 폴더를 직접 열어봐도 동일하게 보인다

## Step 3 — Claude Code 가 이 Vault 를 읽게 만들기

여기서부터가 "AI 가 읽을 수 있는 지식 베이스" 의 핵심입니다. Claude Code 는 폴더의 루트에 있는 `CLAUDE.md` 파일을 **매 세션 시작 시 자동으로 읽습니다.** 이 파일에 "이 Vault 는 무엇이며, 너는 어떻게 행동해야 하는가"를 적어두면, 매번 설명할 필요가 없어집니다.

### 3-1. CLAUDE.md 파일 만들기

Obsidian 의 파일 탐색기에서 **Vault 루트**(맨 위) 에 **우클릭 → "New note"**. 파일명을 `CLAUDE.md` 로 저장합니다.

⚠️ Obsidian 이 기본으로 `.md` 를 붙이므로 그냥 `CLAUDE` 라고만 입력해도 됩니다. 대문자 주의 (`Claude.md` 아님).

### 3-2. CLAUDE.md 내용 붙여넣기

아래를 **그대로** 복사해 붙여넣고, 대괄호 부분만 본인 연구에 맞게 수정합니다:

````markdown
# Research Wiki — 프로젝트 맥락

## 이 Vault 의 목적
이 Vault 는 [내 연구 주제: 예 — "청년층 식품 불안정과 AI 개입의 효과"] 에 대한
[석사/박사] 논문 준비용 개인 지식 베이스이다.

## 폴더 규칙
- `raw/` — 원본 자료 (논문 발췌, 인터뷰 녹취록, 스크랩). **수정 금지, 추가만.**
- `wiki/concepts/` — 개념 하나당 파일 하나. 파일명은 한국어 그대로 (예: `검증부채.md`).
- `wiki/methods/` — 연구 방법론 하나당 파일 하나.
- `wiki/authors/` — 주요 저자·연구자 하나당 파일 하나.
- `papers/` — 논문 1편당 파일 1개. 파일명 형식: `[저자]_[연도]_[핵심키워드].md`
- `drafts/` — 내 글 초안.
- `log/` — 날짜별 작업 기록 (예: `2026-04-21.md`).

## 위키 페이지 작성 규칙
1. 각 페이지는 다음 구조를 따른다:
   - **정의**: 한두 문장으로 무엇인지
   - **출처**: 어디서 읽었는지 ([[논문파일명]] 으로 링크)
   - **나의 연구와의 관련성**: 1~3줄
   - **관련 개념**: [[다른 개념]] 으로 링크
2. 다른 페이지의 개념을 언급할 때는 반드시 `[[위키링크]]` 를 건다.
3. 주장에는 출처를 단다. 출처 불명 내용은 `> [미확인]` 인용으로 표시한다.
4. 페이지는 짧게 유지한다 (1페이지 = 1개념). 길어지면 분리한다.

## AI(너) 에게 기대하는 행동
- 새 자료(`raw/` 나 `papers/` 추가분) 를 읽으면, 관련 `wiki/` 페이지를 찾아 업데이트한다.
- 새 개념이 나오면 `wiki/concepts/` 에 새 파일을 만들고, 기존 페이지에서 `[[링크]]` 를 건다.
- 사용자가 질문하면, 먼저 `wiki/` 와 `papers/` 를 검색해 근거를 찾은 뒤 답한다.
- 근거 없이 추측한 내용은 문장 끝에 `(추정)` 을 붙인다.
- 모든 작업 변경은 `log/YYYY-MM-DD.md` 에 한 줄씩 기록한다.

## 출력 형식
- 한국어로 작성한다.
- 학술적 문체를 유지한다.
- 표를 적극 활용해 비교 정리한다.
````

저장(macOS `Cmd+S`, Windows `Ctrl+S`).

### 3-3. Claude Code 로 Vault 열기

터미널(macOS) 또는 PowerShell(Windows) 을 엽니다. Vault 폴더로 이동 후 Claude Code 를 실행합니다:

```bash
# macOS
cd ~/Documents/research-wiki
claude
```

```powershell
# Windows (PowerShell)
cd $HOME\Documents\research-wiki
claude
```

Claude Code 가 시작되면 첫 줄에 `CLAUDE.md` 를 읽었다는 표시가 뜹니다. 여기서 이렇게 쳐봅니다:

```
> 이 Vault 의 목적과 폴더 규칙을 한 문단으로 요약해줘.
```

AI 가 CLAUDE.md 에 쓴 내용을 기반으로 요약하면 성공입니다. **이 순간부터 AI 는 당신의 Vault 를 "이해" 합니다.**

⚠️ **Claude Code 가 아직 안 깔려 있다면**: 본 워크숍 `hands_on/SETUP.md` 또는 `hands_on/references/claude_code_퀵가이드.md` 를 먼저 참조하세요.

### ✔ 체크포인트

- [ ] Vault 루트에 `CLAUDE.md` 가 있고 내용이 본인 연구에 맞게 채워져 있다
- [ ] 터미널에서 Vault 폴더로 이동해 `claude` 를 실행할 수 있다
- [ ] "이 Vault 의 목적을 요약해줘" 에 대해 AI 가 CLAUDE.md 기반으로 답한다

## Step 4 — 첫 지식 항목 만들기: "검증부채(Verification Debt)"

이론은 충분합니다. 실제 개념 하나를 넣어봅니다. 이 Step 을 다 따라 하면 **나머지 4개는 같은 패턴 반복**입니다.

### 4-1. 파일 생성

Obsidian 파일 탐색기에서 `wiki/concepts/` 폴더 우클릭 → "New note" → 파일명: `검증부채.md`

### 4-2. 내용 작성

아래를 붙여넣고, `[...]` 부분은 내가 읽은 출처에 맞춰 바꿉니다:

```markdown
# 검증부채 (Verification Debt)

## 정의
AI 가 빠르게 생성한 산출물의 품질·진위를 **나중에 몰아서 검증해야 하는 잠재 비용**.
기술부채(technical debt) 의 지식노동 버전.

## 출처
- [[Anthropic_2026_harness-engineering]] p.14
- [[Karpathy_2026_LLM-wiki]] 에서 간접적으로 "knowledge compounding" 개념과 대비로 논의됨.

## 나의 연구와의 관련성
질적 연구에서 AI 가 생성한 코드·테마를 [[근거이론]] 의 교차검증 단계로 흡수할 때,
검증부채가 누적되면 분석 신뢰성이 빠르게 무너진다. 본 연구의 [[방법론 검증 챕터]] 에서
이 개념을 사용하여 파이프라인 설계를 정당화한다.

## 관련 개념
- [[지식복리]] — 검증부채의 반대 방향. 위키 누적으로 검증 비용이 감소하는 현상.
- [[교차검증]] — 검증부채를 줄이는 구체적 방법.
- [[프런티어 모델과 로컬 모델]] — 검증부채가 환경별로 어떻게 달라지는가.

## 메모 (오늘자)
> [미확인] 이 용어가 학술 문헌에 정식 등장한 논문이 있는지 확인 필요.
> 워크숍에서는 실무 용어로 사용 중.
```

저장(`Cmd/Ctrl+S`).

### 4-3. 위키 링크의 마법 확인

위 본문의 `[[지식복리]]`, `[[교차검증]]`, `[[근거이론]]` 은 아직 존재하지 않는 페이지입니다. Obsidian 에서 이 링크들에 마우스를 올리면 **회색**(=존재하지 않음) 으로 보입니다. 클릭하면 **해당 제목의 빈 페이지가 자동 생성**됩니다.

즉, 한 개념을 쓰다 보면 자연스럽게 다른 개념의 껍데기가 생기고, 나중에 채우게 됩니다. 이것이 Obsidian 의 본체입니다.

### 4-4. 그래프 뷰로 확인

왼쪽 사이드바의 **그래프 아이콘**(세 개 점이 연결된 모양) 을 클릭하면, 파일 간 연결이 시각화됩니다. 지금은 `검증부채` 에서 4개의 회색 점이 뻗어나가는 모습이 보입니다. 5개, 20개, 100개 페이지가 쌓이면 이 그래프가 **내 연구의 지도**가 됩니다.

### 4-5. AI 가 읽게 만들기

터미널에서 Claude Code 세션에:

```
> wiki/concepts/검증부채.md 를 읽고, 이 개념과 관련 있을 만한
> 기존 wiki/ 페이지나 논문을 papers/ 에서 찾아줘.
> 없다면 어떤 페이지를 새로 만들면 좋을지 3개 제안해줘.
```

AI 가 답한 제안 중 마음에 드는 것을 `wiki/` 에 페이지로 만들면, Step 4 를 두 번째 반복하는 셈입니다.

### ✔ 체크포인트

- [ ] `wiki/concepts/검증부채.md` 가 존재하고 본인 연구 맥락의 문장이 최소 2줄 들어가 있다
- [ ] 본문에 `[[...]]` 위키 링크가 최소 3개 있다
- [ ] 그래프 뷰에서 `검증부채` 노드가 보인다
- [ ] AI 가 이 페이지를 읽고 관련 제안을 할 수 있다

## Step 5 — 논문 읽을 때와 논문 쓸 때: 전체 사이클

위키는 **축적되어야** 가치가 나옵니다. 여기서는 "어떻게 매일 굴리는가" 를 실제 키 입력 수준으로 설명합니다.

### 5-1. 논문 읽은 직후의 루틴 (10분)

논문 1편을 읽고 PDF 를 덮은 직후:

1. Obsidian 파일 탐색기 → `papers/` 폴더 우클릭 → New note → `[저자]_[연도]_[키워드].md` (예: `Karpathy_2026_LLM-wiki.md`)
2. 아래 템플릿을 붙여넣고 채움:

```markdown
# [논문 제목]

- **저자 / 연도**: Karpathy, 2026
- **출처**: gist.github.com/karpathy/442a6bf555914893e9891c11519de94f
- **1줄 요약**: AI 에게 원본을 매번 재탐색시키지 말고, 미리 컴파일된 위키를 참조하게 하자.

## 핵심 주장
1. 기존 RAG 는 매 질문마다 원본 문서를 재탐색하므로 비효율적이다.
2. 위키는 한 번 컴파일되면 재사용되고 [[지식복리]] 효과를 낸다.
3. 3계층 구조: 원본(raw) / 위키(wiki) / 스키마(CLAUDE.md).

## 인용하고 싶은 문장
> "Knowledge stays compiled, so you don't have to extract it from scratch every time."

## 내 연구에서 어떻게 쓸 것인가
- 서론에서 "연구자의 지식 관리 실패" 문제를 제기할 때 인용.
- 방법론 챕터에서 본 연구가 수집한 자료를 어떻게 위키화했는지 절차를 기술할 때 근거.

## 관련 개념
- [[검증부채]]
- [[지식복리]]
- [[컨텍스트 엔지니어링]]
```

3. 저장.
4. 터미널에서:
   ```
   > papers/Karpathy_2026_LLM-wiki.md 를 읽고,
   > 이 논문이 영향을 주는 wiki/ 페이지를 찾아 업데이트 제안해줘.
   > 새로 만들 개념 페이지가 있으면 어떤 것인지도 알려줘.
   ```
5. AI 가 제시한 업데이트를 검토 후 `y` 로 승인하거나, 내가 직접 편집.

### 5-2. 논문 쓸 때의 루틴 (백링크 활용)

이제 석사 논문 서론을 쓴다고 해봅시다. `drafts/서론.md` 를 열고 씁니다.

문장: "본 연구는 AI 기반 연구 보조 도구의 **[[검증부채]]** 문제를 다룬다."

Obsidian 에서 `[[` 를 입력하면 **자동완성 목록**이 뜹니다. `검증부채` 입력하면 기존 페이지가 추천됩니다. 선택 → `]]` 자동 완성 → 링크 완성.

이제 `검증부채.md` 페이지를 열어보면, 오른쪽 패널에 **"Backlinks"** 가 뜹니다. `drafts/서론.md` 가 이 개념을 참조하고 있다는 것이 자동 표시됩니다.

**이게 왜 중요한가**: 나중에 이 개념의 정의가 정교해지면(Step 4 에서 새 논문을 추가해 페이지가 수정되면), 이 페이지가 인용된 모든 초안을 backlink 로 추적 가능합니다. "아 그 개념을 수정했으니 서론도 손봐야겠다" 가 자동 알림이 됩니다.

### 5-3. 주간 정리 루틴 (토요일 30분)

```
> 지난 일주일간 log/ 에 기록된 변경사항을 리뷰해줘.
> wiki/ 에서 서로 모순되는 주장이나, 출처 없는 주장이 있으면 목록으로 뽑아줘.
> wiki/concepts/ 중 2주 이상 수정되지 않고 backlink 도 없는 "고아 페이지" 가 있으면 알려줘.
```

주 1회 이 작업만 해도 위키가 쓰레기통이 되는 것을 막을 수 있습니다.

### ✔ 체크포인트

- [ ] `papers/` 에 논문 1편이 위 템플릿으로 정리되어 있다
- [ ] `drafts/서론.md` 에서 `[[검증부채]]` 로 링크를 걸어봤다
- [ ] `검증부채.md` 페이지의 Backlinks 패널에 `서론.md` 가 나타난다
- [ ] 주간 정리 프롬프트를 한 번 실행해봤다

## 자주 하는 질문

**Q1. 이미 Zotero 를 쓰고 있어요.**
- 병행하세요. Zotero 는 "논문 저장·인용 관리", 이 위키는 "개념·아이디어 정리". 역할이 다릅니다. Obsidian 플러그인 중 `Zotero Integration` 을 쓰면 Zotero 의 인용키를 이 위키에 자동 가져올 수 있습니다.

**Q2. 노션(Notion) 에 이미 다 있어요.**
- 핵심 개념만 Obsidian 으로 옮기는 것을 추천합니다. 이유: AI 가 읽을 수 있어야 하고, `[[위키링크]]` 와 그래프가 강력합니다. Notion 은 DB·협업에 남기고, 개인 지식 계층만 Obsidian 으로.

**Q3. IRB 민감 데이터를 위키에 넣어도 돼요?**
- 넣어도 되지만, **그 경우 AI 는 반드시 로컬(Ollama) 로** 돌려야 합니다. 클라우드 AI(Claude, ChatGPT) 로는 민감 내용이 포함된 Vault 를 그대로 읽히면 안 됩니다. 별첨 1 (`appendix/01_로컬AI_무료_활용_가이드.md`) 참조.

**Q4. Obsidian 동기화는?**
- 기본 Obsidian Sync 는 유료입니다. 대안: (a) iCloud/OneDrive/Dropbox 의 동기화 폴더를 Vault 로 지정, (b) Git 으로 관리. 연구자에게는 Git 권장 — 변경 이력이 그대로 남습니다.

**Q5. 한국어 파일명 써도 되나요?**
- 네. Obsidian·Claude Code 모두 한국어 파일명을 잘 처리합니다. 단 Git 으로 관리할 경우 macOS 와 Windows 사이에서 파일명 정규화(NFD/NFC) 이슈가 드물게 발생하니, 혼용 환경이면 영어 파일명이 무난합니다.

**Q6. 위키가 너무 커져서 AI 가 다 못 읽어요.**
- 정상입니다. CLAUDE.md 에 "너는 먼저 `wiki/index.md` 를 읽고, 필요한 페이지만 선별해 로드한다" 규칙을 추가하세요. AI 에게 `wiki/index.md` 를 주기적으로 갱신하도록 시키면 됩니다.
